## Summary

Build a fully server-rendered, RTL-first Admin Dashboard for the TEMO manual MLM platform. It adds the `/admin` route group with a sticky sidebar layout, an Overview stats page, Pending Deposits and Tasks approval queues with per-row image-preview modals, a paginated/searchable Users management table with Leadership Level editing (L1–L6), and an indented collapsible Referral Tree per user. All financial mutations are performed via PostgreSQL RPC functions to guarantee atomicity. The middleware and dashboard layout are updated to enforce strict bidirectional role routing. No new npm packages are required.

**Scale/Scope**: 1–5 admin users; ≤10,000 platform users; referral tree depth ≤ 5 levels

## Implementation Phases

### Phase A — Database & Infrastructure

**A1. Migration: Schema additions** (`20260406000009_admin_dashboard_schema.sql`)

1. `ALTER TABLE public.users ADD COLUMN leadership_level SMALLINT NULL CHECK (BETWEEN 1 AND 6)`
2. Add `reviewed_at TIMESTAMPTZ NULL` and `reviewed_by UUID NULL FK → public.users(id)` to both `package_subscription_requests` and `task_completion_logs`
3. Add partial indexes on `status = 'pending'` for both request tables (fast admin queue queries)
4. Add RLS policies granting service-role-only UPDATE access on admin-modified columns

**A2. PostgreSQL RPC Functions** (same migration file)

Five functions, all `SECURITY DEFINER`, all called via `supabaseAdmin.rpc(...)`:

| Function | Description |
|----------|-------------|
| `admin_approve_task(log_id, admin_id)` | Atomic: set log approved + increment wallet_balance + total_earned + insert audit row |
| `admin_reject_task(log_id, admin_id, reason)` | Atomic: set log rejected + store reason + insert audit row |
| `admin_approve_deposit(request_id, admin_id)` | Atomic: set request approved + update user's current_package_level + insert audit row |
| `admin_reject_deposit(request_id, admin_id, reason)` | Atomic: set request rejected + store reason + insert audit row |
| `get_referral_tree(root_id, max_depth)` | Recursive CTE: returns flat `{id, full_name, referral_code, leadership_level, parent_id, depth}` rows |

Each function includes a pending-status guard: raises an exception (caught as a Supabase error) if the record is not in `pending` state — the Server Action maps this to the Arabic race-condition error message.

### Phase B — Middleware & Routing

**B1. Update `middleware.ts`**

Add two rules (in order, before existing checks):
1. `/admin/**` + unauthenticated → redirect `/login`
2. `/admin/**` + authenticated + `role !== 'admin'` (read from `public.users` via JWT custom claim or DB lookup) → redirect `/dashboard`
3. `/dashboard/**` + authenticated + `role === 'admin'` → redirect `/admin`

**Role claim strategy**: The middleware reads the role from the JWT's `app_metadata.role` custom claim (set at user creation via a Supabase Auth hook or seed SQL). This avoids a DB round-trip in middleware. The authoritative check happens in the layout RSC.

**B2. Create `app/admin/layout.tsx`**

- Server Component (no `"use client"`)
- Calls `createClient()`, then `supabase.auth.getUser()`
- Fetches the user's `role` from `public.users` using `createAdminClient()` (service role reads bypass RLS)
- If not admin → `redirect('/dashboard')`
- Renders: `<AdminSidebar>` (desktop) + `<AdminMobileHeader>` + `{children}`

**B3. Update `app/dashboard/layout.tsx`**

Add role check: if `role === 'admin'`, call `redirect('/admin')`.

### Phase C — Server Actions (`app/admin/actions.ts`)

All actions:
1. Call `createClient()` to get the session → validate `role === 'admin'` (reject if not)
2. Validate inputs with Zod (`lib/validations/admin-schemas.ts`)
3. Call `createAdminClient().rpc(...)` to execute the atomic DB function
4. Return `AdminActionResult` (success | error)

```typescript
// Return type shared by all 5 actions
type AdminActionResult =
  | { success: true }
  | { error: string };  // Arabic message for UI display

export async function approveDeposit(requestId: string): Promise<AdminActionResult>
export async function rejectDeposit(requestId: string, reason: string): Promise<AdminActionResult>
export async function approveTask(logId: string): Promise<AdminActionResult>
export async function rejectTask(logId: string, reason: string): Promise<AdminActionResult>
export async function updateUserLevel(userId: string, level: number): Promise<AdminActionResult>
```

`updateUserLevel` does NOT use RPC (no atomicity requirement) — it directly:
```typescript
supabaseAdmin.from('users').update({ leadership_level: level }).eq('id', userId)
```

**Zod schemas** (`lib/validations/admin-schemas.ts`):
- `approveDepositSchema`: `{ requestId: z.string().uuid() }`
- `rejectDepositSchema`: `{ requestId: z.string().uuid(), reason: z.string().min(1).trim() }`
- `approveTaskSchema`: `{ logId: z.string().uuid() }`
- `rejectTaskSchema`: `{ logId: z.string().uuid(), reason: z.string().min(1).trim() }`
- `updateUserLevelSchema`: `{ userId: z.string().uuid(), level: z.number().int().min(1).max(6) }`

### Phase D — UI Components

#### D1. Admin Layout Shell

**`AdminSidebar.tsx`** (Server Component)
- Sticky right sidebar (`fixed inset-y-0 end-0` for RTL)
- Width: `w-64` on desktop, hidden on mobile (`hidden lg:flex`)
- Logo/brand mark at top
- `nav` with 5 links using Next.js `<Link>` + active state detection via `usePathname()` (requires `"use client"` for active highlighting — extract to a thin `SidebarLink.tsx` client wrapper)
- Glassmorphism bottom: user info + logout button

**`AdminMobileHeader.tsx`** (Client Component)
- Sticky top bar visible only on mobile (`lg:hidden`)
- Hamburger icon opens `AdminDrawer`

**`AdminDrawer.tsx`** (Client Component)
- Fixed overlay drawer sliding in from the end (RTL right side)
- Contains same nav links as sidebar

#### D2. Overview Page

**`app/admin/overview/page.tsx`** (RSC)
- Calls `createAdminClient()` three times (or a combined `Promise.all`):
  - `COUNT` of `public.users`
  - `COUNT` of pending `package_subscription_requests`
  - `COUNT` of pending `task_completion_logs`
- Renders 3 `<StatCard>` components

**`StatCard.tsx`** (Server Component, < 50 lines)
- Props: `title: string`, `value: number`, `variant: 'neutral' | 'warning'`
- `warning` variant: amber accent ring + amber text for the number
- Hover: `hover:-translate-y-1 transition-all duration-200`

#### D3. Pending Deposits Page

**`app/admin/deposits/page.tsx`** (RSC)
- Fetches all pending `package_subscription_requests` joined with `users` and `packages`
- For each row, generates a signed URL for `receipt_url` (5-min TTL via `createAdminClient().storage.from('proofs').createSignedUrl(path, 300)`)
- Passes the enriched data array as props to `<DepositsTable>`

**`DepositsTable.tsx`** (`"use client"`)
- Renders a `<table>` with one row per pending request
- Columns: User Name | Phone | Package | Amount | Date | Action (عرض button)
- Clicking "عرض" sets `selectedRow` state → opens `<ProofViewerModal>`
- After approve/reject action: calls `router.refresh()` to re-fetch RSC data

**`ProofViewerModal.tsx`** (`"use client"`, shared between deposits and tasks — place in `app/admin/_components/`)
- Props: `imageUrl`, `title`, `onApprove`, `onReject` callbacks
- Full-screen overlay with centered image viewer
- Image: `<Image>` component (constitution: raw `<img>` forbidden)
- Fallback if image fails to load: Arabic "الصورة غير متاحة" placeholder
- Approve button: `bg-emerald-600`, calls `onApprove()` → loading state during pending
- Reject button: `text-red-600`, reveals inline rejection reason `<textarea>` — Save button disabled until non-empty non-whitespace text entered
- Two-step confirmation: clicking Reject shows the reason input; clicking "تأكيد الرفض" submits

#### D4. Pending Tasks Page

Identical structure to Deposits, but:
- Data source: `task_completion_logs` joined with `users` and `tasks`
- Signed URL for `proof_url`
- Calls `approveTask` / `rejectTask` server actions

#### D5. Users Page

**`app/admin/users/page.tsx`** (RSC)
- Reads `?search` and `?page` from `searchParams`
- Calls `createAdminClient()` with `ilike` filter + `range()` pagination
- Passes `users[]`, `totalCount`, `page`, `search` to client components

**`UserSearchForm.tsx`** (`"use client"`)
- Controlled input + submit button
- On submit: `router.push('/admin/users?search={value}&page=0')`

**`UsersTable.tsx`** (`"use client"`)
- Renders paginated table (20 rows/page)
- Columns: Name | Phone | Package | Balance | Level | Registered | Actions
- Level column: `L{n}` label or "—" if null
- "تعديل" button per row → sets `selectedUser` → opens `<EditLevelPanel>`
- "عرض الشجرة" button per row → navigates to `/admin/users/{id}/tree` or sets `treeUserId` state

**`EditLevelPanel.tsx`** (`"use client"`)
- Fixed slide-over panel from right edge (RTL end)
- Shows user details (read-only)
- `<select>` dropdown: options L1–L6 (values 1–6), preselected to current level (or empty if null)
- Save → calls `updateUserLevel(userId, level)` Server Action → `router.refresh()` on success → toast

**`ReferralTree.tsx`** (`"use client"`)
- Props: `nodes: TreeNode[]` where `TreeNode = { id, full_name, referral_code, leadership_level, children: TreeNode[] }`
- RSC parent (`app/admin/users/[id]/tree/page.tsx`) calls `get_referral_tree` RPC → converts flat array to nested structure → passes as props
- Recursive rendering: each node renders self + (if expanded) its children indented with CSS connector lines
- Toggle state: `Set<string>` of collapsed node IDs (stored in `useState`)
- Root node always starts expanded; children start collapsed
- Leadership level badge: `bg-emerald-50 text-emerald-700 text-xs px-2 py-0.5 rounded-full`
- CSS connector lines: `border-r-2 border-slate-200` on the indent container + `before:` pseudo for horizontal line (Tailwind arbitrary: `before:absolute before:top-4 before:end-0 before:w-4 before:border-t-2 before:border-slate-200`)

## Security Checklist

- [ ] Middleware reads role from JWT, rejects `/admin` without admin claim
- [ ] `app/admin/layout.tsx` re-validates role from DB on every render
- [ ] Every Server Action re-validates `role === 'admin'` from session before any DB op
- [ ] `createAdminClient()` only called inside server-only files (never `"use client"`)
- [ ] Signed URLs generated server-side with 5-minute TTL; raw storage paths never sent to client
- [ ] Rejection reason validated server-side (non-empty, non-whitespace) via Zod
- [ ] RPC functions have pending-state guard to prevent race-condition double-processing
- [ ] `SUPABASE_SERVICE_ROLE_KEY` not referenced in any client component
