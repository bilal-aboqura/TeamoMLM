# Implementation Plan: Investment & Trading Cycles Module

**Branch**: `010-investment-trading-cycles` | **Date**: 2026-04-25 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `/specs/010-investment-trading-cycles/spec.md`

---

## Summary

Allow users to lock capital (USDT) into a tiered investment account and earn fixed-percentage profits every 7 days, calculated lazily on-read with no cron jobs. Admins review deposit receipts and withdrawal requests through a single tabbed panel. The profit engine is a pure computation: `floor((now - cycleStart) / 7days) × floor(capital × tier% / 100, 2 dp)`.

**Scale/Scope**: Single investment per user; 5 fixed tiers; manual admin review

## Phase 0: Research — Resolved

All research findings in [research.md](./research.md). Key decisions:

| Decision | Resolution |
|----------|-----------|
| Profit calculation | Lazy on-read; `floor((now - cycleStart) / 7days) × floorProfit` |
| Rounding | `Math.floor(value * 100) / 100` — floor to 2 decimal places |
| Cron jobs | None — lazy computation only |
| Storage | New private `investment-receipts` bucket; mirrors `equity-receipts` pattern |
| Admin mutations | All via PostgreSQL SECURITY DEFINER RPCs for atomicity |
| Tier config | Hardcoded TypeScript constant in `lib/investment/tiers.ts` |
| Notifications | New `in_app_notifications` table introduced by this feature |
| Concurrent withdrawal prevention | RPC uses `FOR UPDATE` row lock + inline `available_profit` check |
| Cycle termination | Indefinite — `Completed` only via explicit admin/user action |
